# 质量评审标准

## 目标

本文件定义 batch reviewer 的统一评审口径。
reviewer 的职责只有两件事：

- 判断这一批是否达到可发布水平
- 判断是否存在必须阻断发布的结构性问题

reviewer 不负责逐句润色，也不负责提出“还能不能更好”的开放式建议。
优先看结构、功能、连续性、越界和明显退化。

## 输入原则

具体要读哪些文件，由每个 batch 的 reviewer prompt 提供。
本标准只定义“怎么判”。

## Verdict 矩阵

### 必须 PASS 的条件

同时满足以下条件，才能给 `PASS`：

- 本批关键 beats 已完成，没有明显漏掉本集主戏
- 抽检集没有 blocking 级结构问题
- 没有明显偷走后续集 payoff
- 没有明显 source 越界或新增关键剧情
- 成稿人物姓名已完成改名，没有沿用原著主要人物、反派或关键配角姓名
- 抽检集没有明显角色知识偷跑；称谓、姓氏、全名、职位、亲属关系和身份判断能在正文前文或 `knowledge_boundary` 中找到来源
- 批次内角色弧光、冲突推进、关系顺序基本连贯
- 如果抽检包含 EP-01 到 EP-03，开场已满足短剧黄金开场：快速进入异常、冲突、羞辱、逼选、误认、证据或身份错位
- 核心场没有明显停留在解释、资料宣读或功能性对白层面；关键信息已外化为动作、站位、阻力、选择或现场后果

### 必须 FAIL 的条件

出现以下任一情况，必须给 `FAIL`：

- 抽检集缺失核心 beat 或核心主戏
- 当前集新增了会改变原著意图的重要剧情
- 当前集提前消费了本应留给后续集的揭露、和解、反打、打脸或情绪 payoff
- 成稿沿用了原著主要人物、反派或关键配角姓名，或只做了同音/近音/只改姓/只改名的弱改名
- 核心冲突、误认、认亲、掉马、逼选或集尾钩子中出现明显角色知识偷跑，导致人物说出当场不该知道的姓氏、全名、职位、关系或身份结论
- 批次内角色立场、关系推进、冲突顺序出现明显断裂
- 与 `quality.anchor.md` 相比，在结构厚度、冲突重量、场尾推进上出现明显退化
- 证据、资料、屏幕或录音承担了本集核心反转，但现场没有人物选择、阻力、空间动作或后果落点，导致主戏像汇报材料而不是可拍场面
- 抽检集的核心主戏主要由解释句、功能性台词或资料宣读撑起，删掉这些文字后没有可拍动作链或人物选择链
- EP-01 开场明显慢铺，前 10-20 秒仍在环境、小传、解释或等待事件发生，未建立核心异常/冲突/逼选
- 前 3 集只是在重复同一种误认、羞辱、盘问或拉扯，没有逐集升级

### 只能记为 WARNING 的情况

以下问题默认只记 `warning`，不应单独导致 `FAIL`：

- 首集厚度略保守，但主戏仍成立
- 个别 `os` 略直给，但未偷解释、未替代剧情
- 局部台词或场次还有模板味，但不影响主戏功能
- 个别非核心台词略别扭或称谓来源不够清楚，但不影响主戏理解和关键关系推进
- 批次内有轻微风格波动，但角色和推进仍可辨认
- 证据、资料、屏幕或录音推进偏多，但仍有明确现场阻力、人物选择和结果动作
- 个别场面仍有解释句，但核心冲突已由动作、站位、沉默、阻力或现场后果承载
- EP-02 或 EP-03 开场冲突不够锋利，但本集主戏、升级和钩子仍成立

如果 warning 已经实质影响主戏成立、功能顺序或发布可信度，就必须升级为 `FAIL`。

## 必做评审

### 1. 抽检集结构评审

对所有 sampled episodes 至少完成以下检查：

- 角色替换测试
  如果关键对话换给别的角色也不违和，说明角色声纹或功能分配有问题。
- 人名改编测试
  对照 source、blueprint 或 source.map，检查剧中主要人物、反派、关键配角是否已彻底改名；改名必须不破坏角色关系和 source 事件顺序。
- 角色知识边界测试
  对照 `knowledge_boundary` 和正文前文，检查关键称谓、姓氏、全名、职位、亲属关系、婚恋关系和身份判断是否有现场来源。source.map 里的全名只算编剧定位，不自动算角色已知。
- 关键台词顺口度测试
  抽查每集第一场、核心反击句和集尾钩子。若出现错别字、病句、主谓宾缺失或真人不会这么说的夹生句，按影响程度记 warning；如果发生在核心反击或钩子并破坏理解，可升级 blocking。
- 删除测试
  如果删掉某场戏后，本集主戏几乎不变，说明这场戏可能是结构性空转。
- 逻辑反推测试
  如果无法从前文因果合理推出该场结果，说明结构支撑不足。
- 钩子有效性测试
  如果集尾没有明确前推压力，只是“停在一个结束点”，说明集尾功能偏弱。
- 黄金开场测试
  对 EP-01 到 EP-03，检查第一场第一组动作或第一句台词是否已经进入异常、冲突、羞辱、逼选、误认、证据或身份错位；如果仍在铺环境、介绍人物或复述前情，应记 warning 或 blocking。
- 目标总时长节奏测试
  当前 batch 是否符合 `run.manifest.md` 中 `target_total_minutes` 对应的高密度完整剧集节奏：每集有主戏闭环，每 3-5 集能看到阶段性释放，而不是百集付费模型式拖长。
- 画面感测试
  如果核心场无法稳定想象成可拍画面，且影响剧情功能，应记 warning 或 blocking。
- `os` 有效性测试
  如果 `os` 只是在重复表层信息，没有提供隐藏判断、代价感或反差，应记 warning。
- 证据戏外化测试
  如果删掉屏幕、文件、录音文字后，只剩人物站着听资料，本场不合格。至少要有其一：抢夺、阻拦、封门、按下遥控器、换站位、扣手铐、签字、冻结权限、交接证物、谁必须当场选择。
- 行动承载测试
  核心信息必须绑定一个现场动作和一个人物立场变化。证据只能证明事实，不能替代冲突。
- 自我打磨完成度测试
  判断成稿是否像经过 agent polish：解释句是否被动作替代，功能性台词是否被压缩，核心场是否至少有 1-2 个压力点或选择点，角色声纹是否仍可区分。

### 2. 批次级连续性评审

对整个 batch 至少完成以下检查：

- 角色弧光连续性
  人物态度、立场、关系推进不能前后打架。
- payoff 顺序检查
  当前集不能偷走后续集才该释放的功能或情绪收益。
- 质量锚对标
  对照 `quality.anchor.md`，判断是否出现结构性退化，而不是单纯文风波动。
- open loop 连续性
  已埋下的压力、悬念、关系线，必须仍能看到后续承接路径。

## 输出契约

reviewer 只能通过以下字段回填结论：

- `status`
- `reviewer`
- `reason`
- `blocking_reasons`
- `warning_families`
- `arc_regressions`
- `function_theft_findings`
- `quality_anchor_findings`
- `evidence_refs`

不要额外发明 schema。

## 证据规则

每个 blocking 结论至少提供一条具体 `evidence_ref`。

推荐写法：

- `draft:EP-03:scene2`
- `draft:EP-05:lines 28-44`
- `source:chapter3`
- `brief:EP-04`

## Reviewer 纪律

- 不要直接重写剧本
- 不要把 review 做成逐句 copy edit
- 不要因为“还能更好”就随意给 `FAIL`
- 对结构缺失、payoff 偷窃、source 越界、弧光断裂保持严格
- 对“资料宣读替代戏剧动作”保持敏感；如果它承载核心主戏，优先按结构问题处理，不要只记文风问题
