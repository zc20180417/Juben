# 写作合同（Write Contract）
<!-- PROMPT-REFERENCE: Writer prompt may reference stable contract sections below. Keep section ids stable when editing. -->

本文件定义 writer 的通用写作边界，不包含项目专有章节映射。
具体写法风格补充见 `harness/framework/writer-style.md`。
本文件优先定义硬门槛、可执行边界和尽量可验证的规则；若与 `writer-style.md` 冲突，以本文件为准。

## 写作姿态（Writing Posture）
<!-- SECTION:WRITING_POSTURE -->
- `△` 只写摄影机能拍到的内容
- `os` 是主观窗口，但不能承担剧情讲解职责
- 完整气口优先于碎句节拍（见下方“对白排版（Dialogue Formatting）”）
- 具体场景打法、叙事姿态与风格警戒统一遵循 `writer-style.md`

## 标记格式（Marker Formatting）
<!-- SECTION:MARKER_FORMAT -->
- `△`、`♪`、`【镜头】：`、`角色：`、`角色（os）：` 各自独占一行，不与其他标记混排
- `△` 只承担可见动作、环境、道具、姿态和物理细节
- `♪` 只承担环境声、音效、节奏停顿、配乐切入或刻意沉默
- `角色（os）：` 必须带明确角色名，不写成匿名念头
- `♪` 与 `【镜头】：` 是壳层语法，不是装饰；只有在新增节奏、视角或信息落点时才写，不能机械铺满

## 对白规则（Dialogue Rules）
<!-- SECTION:DIALOGUE_RULES -->
- 允许换说法，不允许换态度
- 不得把受压、求生、陈述改成挑衅、机锋、评判或聪明压人
- 保留回应句时，必须保留上句触发语义
- 不得把任何角色统一压成碎句机器（包括主角）
- 原著对应段若以施救、守夜、观察、潜伏、追踪为主，禁止为撑节奏主动补密集对白、机锋、狠话或关系宣告句
- 原著若只发生一次短试探或短问答，不得擅自扩成多轮连续盘问；能在一场里完成的，不扩成“户籍式”追问链
- 一句台词若只是在重复镜头里已成立的动作判断，且删掉后场意不变，应删
- 角色尚未走到那一步时，不得替他提前说出关系、亏欠、信任或心软的结论句

## 对白排版（Dialogue Formatting）
<!-- SECTION:DIALOGUE_FORMATTING -->
"短句"指角色说话内容简洁，不是剧本排版上每行都要断开。区分方式：

- **一个语义单元 = 一行**。角色一口气说完的话，即使由几个短句组成，也写在同一行
- 只有在语义上有**停顿、转折、施压**时才换行——换行本身就是节奏工具
- 错误示范（机械断行）：
  ```
  萧景珩：让朕的嫔妃当众献舞。
  萧景珩：你当她是歌姬？
  ```
- 正确示范（语义换行）：
  ```
  萧景珩：让朕的嫔妃当众献舞，你当她是歌姬？
  ```
- 当角色**刻意施压、逼问、一句一句钉**时，才拆成多行：
  ```
  萧景珩：聚聚。
  萧景珩：聚到她掉眼泪？
  ```
  ↑ 这里"聚聚"是冷重复，第二句是反问施压——两句之间有刻意停顿，所以分行。
- 判断标准：如果拆开后两行之间**读不出停顿感**，就不该拆

## `OS` 规则（OS Rules）
<!-- SECTION:OS_RULES -->
- `os` 必须一眼读清主语、对象和判断落点
- `os` 不得悬空到需要读者自己补半句
- `os` 不解释剧情，不提前宣布计划得手，不把观众判断说穿
- `os` 应优先承担：表里不一、代价感、被触发的旧记忆
- 删掉后若场意、人物动机和关系变化都仍然成立，该 `os` 应删
- `os` 不得把角色尚未承认的关系判断、情绪结论或观众本该自己感受到的变化直接说透

## 场次规则（Scene Rules）
<!-- SECTION:SCENE_RULES -->
- 每集至少 2 场；场次数由功能完成决定，不设置硬上限，但不得用空场凑时长
- 每集必须有明确戏剧推进；“推进”至少落在其一：关系位移、新信息、目标变化、新威胁、决定、代价
- 一场只主打一项功能；若同一场同时承担两个强功能，应拆场而不是糊成总结场
- 集尾要么有前推力，要么有强闭环落点
- 即时动作钩子只有在相邻下一集第一场会兑现时才允许使用
- 相邻两集连续承接同一时间地点时，下一集必须尽快进入新增推进，不得把上一集尾部已完成动作整场重演
- 低对白场允许主要由动作、停顿、视线、道具和位置变化承担推进，不强制补对白填满
- 场景开头的环境交代只保留最有效的 1-2 个视觉 / 听觉符号；若删掉一半晨光、残滴、白气、风声后信息不变，说明写密了
- EP-01 到 EP-03 执行黄金开场：第一场第一组可拍动作或第一句台词必须直接进入异常、冲突、羞辱、逼选、误认、证据或身份错位，不得慢铺人物小传
- EP-01 必须在本集内完成“异常入局 + 第一轮关系冲突 + 强钩”；不得只写事件前夜或停在风暴将起

## 改编姿态（Adaptation Posture）
<!-- SECTION:ADAPTATION_POSTURE -->
- 改编不是压缩字数，而是重分配信息载体
- 当前目标总时长以 `run.manifest.md` 的 `target_total_minutes` 为准；应按该总时长对应的高密度完整剧集理解，不按 80-100 集付费长链条灌水
- 剧中人物姓名必须与原著姓名完全不同；不得沿用原名，不得只改姓或只改名，不得用同音/近音/叠字变体规避
- 改名只改变称呼，不改变人物功能、关系、弧光和 source 顺序
- 若 batch brief、source.map、character.md 或 voice-anchor.md 仍出现原著姓名，writer 应按已给出的改名映射统一替换成剧名；没有映射时必须先在草稿开头外的内部判断中确定映射，但不得把映射说明写进剧本文本
- 先保主线推进，再保关键关系变化，再保爆点与反转，最后保世界规则与主题表达
- 原著叙述信息优先外化成：动作、停顿、道具、环境变化、对话反应
- 第一人称原著默认改写成第三人称可拍视角，不直接保留“我”叙述腔
- 角色心理先尝试外化成表情、身体动作、选择和说话方式；只有画面承载不了时才进入 `os`
- 原著里”写出来很好看”的段落，不等于搬成剧本就有效
- 剧本语言只保留人物会说的话、摄影机能拍到的事、剪辑能组织出的信息
- 原著叙述腔、优美句子、哲理句、旁白总结句不直接入正文
- `△` / `vo` / `os` 不直接沿用小说式“我如何看见、我如何觉得、我如何走进去”的叙述句
- 当原著气口与“短剧补词”冲突时，优先保原著气口；宁留白，不补模板狠话
- 原著低对白源段，不得硬改成高交锋、高机锋或高表态场
- 原著若把压力主要放在某一方“被看、被试、怕暴露”的感受上，改编时优先保护这条单侧紧张线，不急着写成双方势均力敌的对手戏
- 身份、姓氏、名号若在原著里天然带分量，剧本里要用可感的联想或后果把重量落地，不能只写成抽象氛围

## 声纹锚点（Voice Anchor）
<!-- SECTION:VOICE_ANCHOR_USAGE -->
- writer 优先读取 `voice-anchor.md`
- 若 `voice-anchor.md` 不存在、仅剩模板占位或未列出当前角色，回退到 `character.md` 中的语音指纹
- 只继承说话方式，不搬运小说叙述腔
- 角色台词必须符合声纹锚中的句长基准、句式基准和禁用表达

## 成稿风格（Writer Style）
<!-- SECTION:STYLE_HANDOFF -->
- 叙事立场、场景打法、对话打法、集级节奏和风格警戒统一定义在 `writer-style.md`
- contract 负责门槛，`writer-style.md` 负责稳定成稿时的具体执行方式
- 两者冲突时，以本文件的边界与硬门槛为准

## 原著保真规则（Original-Fidelity Rules）
<!-- SECTION:ORIGINAL_FIDELITY -->
- 章节顺序优先
- 人名不属于本项目的保真对象；保真的是关系、功能、事件顺序和情绪逻辑
- 一章可拆多集，多个相邻章节可合拍
- 不得抢跑后续章节大事件
- 原著自然发生的关键节点，不得无依据改成预谋布局
- 原著概述性信息，不得擅自扩成新的人名、具体事故或完整事件链
- 若原著某场核心是”某个人第一次单独注意到她 / 记住她 / 被她刺中”，不得扩成多人围观、多人评价的群像场
- 这类单点被看见、单点被记住、单点被刺中的关系场，优先保护其专属性
- 原著若只给一轮简短试探，优先保“短”和“准”；不得为补时长把它改写成多场连续问身世、问来处、问家门的流程场

## 交稿前自检（Pre-Submit Check）
<!-- SECTION:PRE_SUBMIT_CHECK -->
- 检查成稿中是否出现原著人物姓名；如出现，必须替换成剧名
- 若当前集是 EP-01 到 EP-03，检查开场前 10-20 秒是否已经进入强冲突或强悬念；若还在铺环境、介绍人物或等待事件发生，必须重写开场
- 检查本集是否有主戏闭环：进入、对抗/变化、结果或钩子；若只是几个事件点顺排，必须重组
- 当前集若改编自低对白源段，检查是否被写成“靠台词撑戏”；若是，先删台词，再补动作
- 检查是否出现只为立人设而写的冷句、狠话、机锋句；若删掉后人物仍成立，则删
- 检查 `os` 是否重复了画面、台词或观众已能感受到的信息；若重复，则删
- 检查试探场是否被扩成多轮盘问；若删去一半问答后主信息仍成立，说明写多了，应压短
- 检查压力是否被错误写成“双方都很会试探”的均衡对戏；若原著核心其实是谁更怕暴露，就把镜头和感受重新压回那个人
- 检查开场环境是否在堆同层氛围词；若雨声、灯影、白气、晨光删去一半后场意不变，应删
- 检查 `♪` / `【镜头】` 是否只是重复已成立画面；若不承担新增信息或节奏转折，应删
- 检查集尾是否有新增推进，而不是仅停在情绪余波或平地对视

## 角色知识边界（Character Knowledge Boundary）
<!-- SECTION:CHARACTER_KNOWLEDGE -->
- 作者已知信息、batch brief 信息、source map 信息，不自动等于角色当下已知信息
- 角色台词、`os`、动作说明和镜头说明，都只能使用当前场内已经听到、看到、被明示或按关系天然知道的信息
- `source.map` / batch brief 中的全名只是编剧定位，不等于角色已经知道对方全名
- 每集 `knowledge_boundary` 是本集称谓和身份信息的执行边界，必须先读再写台词
- 只听到名，不得主动补姓；只见到脸，不得直接说全名；只知道“她 / 那位小姐 / 那个人”，不得偷换成更具体的身份称谓
- 第一次使用姓氏、全名、职位、亲属关系、婚恋关系或身份名号，必须能在正文前文找到来源：自我介绍、旁人介绍、名牌、文件、屏幕、电话备注、公开称呼或天然旧关系
- 如果正文找不到来源，就退回中性称谓；不要为了台词顺手提前喊“顾先生”“姐姐”“未婚夫”“总裁”等关系/职位词
- 原著若把姓名、身份、关系词拆成前后两步解锁，剧本也必须按原顺序解锁；不得把后段信息偷跑到前段台词里
- 对角色知识边界拿不准时，优先使用更中性的称谓与回应，不替角色抢答
- 回应句必须接住上一句真实触发词；上一句没出现的称谓、关系词和指控，不得凭作者脑内上下文自行补全
